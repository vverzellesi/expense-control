# Brainstorming Ideas Into Designs

## Overview

Help turn ideas into fully formed designs and specs through natural collaborative dialogue.

Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design in small sections (200-300 words), checking after each section whether it looks right so far.

## The Process

**Understanding the idea:**
- Check out the current project state first (files, docs, recent commits)
- Ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible, but open-ended is fine too
- Only one question per message - if a topic needs more exploration, break it into multiple questions
- Focus on understanding: purpose, constraints, success criteria

**Exploring approaches:**
- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation and reasoning
- Lead with your recommended option and explain why

**Presenting the design:**
- Once you believe you understand what you're building, present the design
- Break it into sections of 200-300 words
- Ask after each section whether it looks right so far
- Cover: architecture, components, data flow, error handling, testing
- Be ready to go back and clarify if something doesn't make sense

## After the Design

**Documentation:**
- Write the validated design to `ai_docs/brainstorm/YYYY-MM-DD-<topic>-design.md`
- Commit the design document to git

**Implementation (if continuing):**
- Ask: "Ready to set up for implementation?"
- Use writing-plans to create detailed implementation plan, you must send the absolute path to the brainstorming document

**Exceção:** Código, nomes de variáveis, e termos técnicos em inglês mantêm sua grafia original.

## Key Principles

- **One question at a time** - Don't overwhelm with multiple questions
- **Multiple choice preferred** - Easier to answer than open-ended when possible
- **YAGNI ruthlessly** - Remove unnecessary features from all designs
- **Explore alternatives** - Always propose 2-3 approaches before settling
- **Incremental validation** - Present design in sections, validate each
- **Be flexible** - Go back and clarify when something doesn't make sense

# Writing Implementation Plans

## Overview

Create detailed, actionable implementation plans through interactive research before writing code.

**Core principle:** A plan is a specification that enables mechanical implementation. If implementation requires invention or decision-making, the plan has failed.

**This skill is a WORKFLOW.** Execute it step-by-step. Every step has explicit transitions. NEVER skip steps. NEVER invent your own order.

## When to Use

Use when ANY of these apply:
- Starting work on a new feature or ticket
- Task requires understanding existing patterns before coding
- Requirements span multiple files or systems
- You need to coordinate phases of work
- Success criteria need to be explicit

## When NOT to Use

Skip when:
- Simple bug fix with obvious solution
- Task is well-understood and can be completed quickly
- User explicitly wants code, not a plan

---

## The Five Iron Laws

### 1. No Plan Without Research

```
NO PLAN WITHOUT CODEBASE RESEARCH FIRST
```

Skip research? Your plan is worthless. Start over.

### 2. Plans Contain Code

```
THE PLAN IS THE CODE, JUST NOT YET IN FILES
```

If implementing requires thinking/designing, the plan is incomplete.

### 3. Tests Are Not a Separate Phase

```
EACH PHASE INCLUDES ITS TESTS
```

A phase without its tests is NOT shippable.

### 4. Phase Count Matches Complexity

```
SMALL CHANGES = SINGLE PHASE
```

Would a senior engineer split this into multiple PRs? NO → single phase.

### 5. 100% Rule (No Gaps, No Overlaps)

```
PLAN CAPTURES ALL WORK - NOTHING MORE, NOTHING LESS
```

---

## Execution Model: Know Your Consumer

Your plan is consumed by `implementing-plan`. Understanding its execution chain shapes how you write plans.

### Per Phase:
1. ONE `implementation-executor` subagent dispatched (200K context window)
2. Executor reads phase file + all `Load Before Implementing` files
3. Executor runs TDD per task: RED → GREEN → REFACTOR
4. Executor produces structured **Phase Execution Report** (validated by orchestrator)
5. On validation failure: re-dispatch with error context (max 3 attempts → user escalation)
6. `git commit --no-verify`
7. `Skill(running-automated-checks)`
8. README dashboard updated (phase status + learnings)

### After ALL Phases:
9. `Skill(reviewing-code)` — mandatory, non-negotiable
10. `Skill(validating-implementation)` — mandatory, non-negotiable

### What This Means For Plan Writing

| Constraint | Implication |
|------------|-------------|
| **≤3 tasks per phase** | Hard limit. More tasks = context overload. Split into more phases instead. |
| **One executor per phase** | Phase = dispatch unit. Phase sizing determines executor workload. |
| **Plan code is executed verbatim** | Wrong imports, missing functions, bad file paths → re-dispatch. |
| **Executor has 200K context** | Phase file + all referenced files + TDD code must fit. |
| **TDD: YES/NO feeds validation** | Executor report includes per-task TDD results. Mark accurately. |
| **Max 3 re-dispatches** | After 3 failures, user gets escalated. Write plans that succeed first try. |
| **Review + validation are mandatory** | Plan code must be review-ready, not just "works." |

## The Workflow

Execute these steps IN ORDER. Every step has explicit transitions.
NEVER skip a step. NEVER proceed past ⛔ without user response.

### STEP 1: Read Mentioned Files

**WHY:** Build initial context from what the user already provided.

- Read ALL mentioned files COMPLETELY (tickets, research docs, related plans)
- Use Read tool WITHOUT limit/offset
- NEVER read partially

→ Go to **STEP 2**

### STEP 2: Research

**WHY:** Understand the codebase before planning. `researching-codebase` produces an optimized, structured document.

- Invoke `researching-codebase` via subagent:

  ```
  Task(
    subagent_type: "general-purpose",
    description: "Research [FEATURE/SYSTEM]",
    prompt: "Use Skill(researching-codebase) to research how [FEATURE/SYSTEM] currently works in preparation for [TASK]. Save findings to ai_docs/research/YYYY-MM-DD-description.md. Return summary of key findings."
  )
  ```

- Read the research document COMPLETELY in main context
- DO NOT re-read raw files the research already covers

→ Go to **STEP 3**

### STEP 3: Research Quality Gate

**WHY:** Insufficient research produces plans that fail during implementation.

Answer ALL 7 questions:

1. What files will I modify and why?
2. What existing code does something similar?
3. What patterns/conventions must I follow?
4. What will break if I get this wrong?
5. How is similar code tested in this codebase?
6. What related systems exist that I'm NOT changing but might affect understanding?
7. Have I traced the full data flow lifecycle?
   - When does each component initialize?
   - When does config/data flow from source to consumer?
   - Are there timing dependencies between components?

**Branching:**

- ALL 7 answered → Go to **STEP 4**
- SPECIFIC file detail missing → Read that ONE file directly → Go to **STEP 3**
- BROAD gap remains → Re-invoke `researching-codebase` with targeted query → Go to **STEP 2**
- Need USER clarification → AskUserQuestion → incorporate answer → Go to **STEP 3**

### STEP 4: ⛔ Present Findings

**WHY:** User must confirm understanding before structure work begins.

Present using this format:

```
Based on [task], I understand we need to [summary].

I've found that:
- [Current implementation detail with file:line]
- [Pattern or constraint discovered]

Questions that my research couldn't answer:
- [Technical question requiring judgment]
- [Business logic clarification]

Ready to proceed to structure development?
```

**⛔ STOP. Wait for user response.**

**Branching:**

- User confirms → Go to **STEP 6**
- User corrects or adds info → Go to **STEP 5**

### STEP 5: Verify Corrections

**WHY:** User corrections must be verified against actual code. DO NOT accept blindly.

- Re-invoke `researching-codebase` with targeted follow-up query
- Read new research document
- NEVER just accept a correction without verification

→ Go to **STEP 3**

### STEP 6: Propose Structure

**WHY:** Get alignment on scope and phasing before writing details. Phasing is the most important structural decision — it determines how the executor consumes the plan.

- Determine phase count using Iron Law #4
- **Hard limit: ≤3 tasks per phase.** The executor processes one phase in a single context window. More tasks = more files to load + more TDD cycles = context exhaustion. Prefer more phases with fewer tasks over fewer phases with many tasks.
- Phase count = ceil(tasks / 3). 6 tasks → 2 phases, 9 tasks → 3, 12 tasks → 4, 15 tasks → 5. Do NOT default to any fixed number.

#### How to Phase Well

**The goal:** Each phase produces a working, demonstrable increment. Phase 1 should produce something you can run and see working — not just a library or a schema.

**The Tracer Bullet Principle applied to phases:**
- Phase 1 = end-to-end skeleton. A minimal working system touching ALL layers.
- Subsequent phases = expand capabilities on top of the working skeleton.

**Phase Quality Checklist:**
1. **Working increment** — After this phase, can you demo something? If not, merge it with the next phase.
2. **Vertical, not horizontal** — Each phase touches all necessary layers (types → logic → API → test), not one layer at a time.
3. **Natural boundary** — Phase boundary aligns with a commit/PR point. Would a senior engineer pause here for review?
4. **Minimal coupling** — Later phases don't force changes to earlier phases. Dependencies flow forward.
5. **Tests included** — Each phase has its own tests. No "tests phase" at the end.

**Common phasing mistakes:**

| Bad Phasing | Why It Fails | Better Phasing |
|-------------|-------------|----------------|
| P1: All types/interfaces, P2: All logic, P3: All tests | Horizontal layers. Nothing works until P3. | P1: One feature end-to-end with types+logic+tests. P2: Next feature. |
| P1: Core library, P2: API using library | P1 has no demo. You can't run it. | P1: Library + one API endpoint + test = runnable. P2: Remaining endpoints. |
| P1: Backend, P2: Frontend | Nothing user-facing until P2. | P1: One user flow end-to-end. P2: Additional flows. |
| P1: Setup/scaffolding, P2: Real work | P1 delivers zero value. | Merge scaffolding into P1 as first tasks. |

**Task budget per phase:**
- **Hard limit: ≤3 tasks per phase.** Non-negotiable.
- One `implementation-executor` handles ALL tasks in the phase (200K context window)
- 3 tasks × ~3 files each = ~9 files loaded. Leaves ample room for TDD cycles.
- If you're tempted to add a 4th task, create a new phase instead

For **single-phase plans** (most common):

```
Here's my proposed plan structure:

## Overview
[1-2 sentence summary]

## Implementation (single phase)
- [File 1]: [change summary]
- [File 2]: [change summary]
- Tests: [what tests]

This is a small, focused change - single phase is appropriate.
Does this make sense?
```

For **multi-phase plans** (only when justified):

```
Here's my proposed plan structure:

## Overview
[1-2 sentence summary]

## Why multiple phases:
[Explain why this can't be a single phase]

## Implementation Phases:
1. [Phase name] - [what it delivers as a working increment]
2. [Phase name] - [what it adds on top of Phase 1]
... (as many phases as the task count requires — do NOT default to any fixed number)

Phase 1 produces: [what you can run/demo after Phase 1]

Does this phasing make sense?
```

→ Go to **STEP 7**

### STEP 7: ⛔ Structure Approval

**WHY:** Writing a full plan for the wrong structure wastes everyone's time.

**⛔ STOP. Wait for user response.**

**Branching:**

- User approves → Go to **STEP 8**
- User adjusts → Modify structure → Go to **STEP 7**

### STEP 8: Write Plan

**WHY:** The plan document enables mechanical implementation.

1. Check for existing directory/file collision
2. Create directory `ai_docs/plans/YYYY-MM-DD-description/`
3. Write README.md — load [references/readme-template.md](./references/readme-template.md)
4. Write phase-N files — load [references/plan-template.md](./references/plan-template.md)
5. Every task uses the required task template format

**Rules during writing:**

- **Greenfield Calibration:** High complexity = full code, Medium = intent + signatures, Low/Trivial = inline in parent task
- **Tracer Bullet:** first task in each phase = end-to-end skeleton for that phase's scope
- **YAGNI:** only build what the first case exercises
- **Trivial tasks** (< 5 lines, zero decisions) → inline in parent task, NOT separate tasks
- **Phase Quality:** Re-verify each phase delivers a working increment (see STEP 6 checklist). If a phase only produces types or scaffolding with nothing runnable, merge it into the next phase.

→ Go to **STEP 9**

### STEP 9: Quality Gates

**WHY:** Quality gates catch plan deficiencies before implementation.

Load [references/quality-gates.md](./references/quality-gates.md) for full checklists.

Run ALL gates:

1. **Template Compliance Gate** — all required plan sections present
2. **100% Rule Verification** — no gaps, no overlaps, all tasks complete
3. **Implementation-Ready Verification** — copy/paste ready, no placeholders
4. **Phasing Quality Gate** — each phase is a vertical working increment, not a horizontal layer
5. **Executor Compatibility Gate** — dependency completeness, no dead references, context budget

**Branching:**

- ALL gates pass → Go to **STEP 10**
- ANY gate fails → Fix the plan → Go to **STEP 9**

### STEP 10: Self-Check

**WHY:** Catch issues before presenting to user and Oracle.

Verify ALL:

- [ ] Did I examine actual code files (not just docs)?
- [ ] Can I cite specific files for each pattern I'm using?
- [ ] Does my plan contain actual implementation code?
- [ ] Can implementation be done by copy/paste without thinking?
- [ ] Did I complete the 100% Rule verification?
- [ ] Does every task have Load Before Implementing listed?
- [ ] Is every acceptance criterion in Given-When-Then format?

**Branching:**

- ALL yes → Go to **STEP 11**
- ANY no → Fix → Go to **STEP 9**

### STEP 11: ⛔ Present Plan

**WHY:** Oracle reviews BEFORE user sees the plan. This catches issues humans miss.

1. Chain to `reviewing-plan` skill (Oracle reviews)
2. Evaluate and apply valid Oracle suggestions
3. Present final plan to user

**⛔ STOP. Wait for user response.**

**Branching:**

- User accepts → **DONE**
- User has feedback → Go to **STEP 12**

### STEP 12: Iterate

**WHY:** User feedback must be incorporated and re-validated.

- Apply user feedback
- If feedback requires new research → Go to **STEP 2**
- If feedback is structural → Go to **STEP 6**
- If feedback is about plan details → Fix → Go to **STEP 9**

# Reviewing Plan

## Overview

After a plan is written, invoke Oracle (GPT-5.2) to review it. Treat Oracle's output as a **junior review** - valuable input that requires senior evaluation before applying.

## When to Use

This skill is automatically chained from `writing-plans`. Do not invoke directly unless reviewing an existing plan.

## Workflow Checklist

Copy and track progress:

```
Plan Review Progress:
- [ ] Step 1: Spawn Oracle subagent with plan file
- [ ] Step 2: Evaluate each Oracle suggestion
  - [ ] Valid + important → Apply
  - [ ] Valid but minor → Apply if quick
  - [ ] Incorrect assumption → Discard
  - [ ] Over-engineering → Discard
  - [ ] Already covered → Discard
- [ ] Step 3: Update plan with valid suggestions
- [ ] Step 4: Present final plan to user
```

## The Iron Law

```
NO PLAN FINALIZED WITHOUT ORACLE REVIEW
```

**But also:**

```
NO ORACLE SUGGESTION APPLIED WITHOUT EVALUATION
```

Oracle is smart but lacks context. Evaluate every suggestion.

## Process

### Step 1: Spawn Oracle Subagent

Use the Task tool with `subagent_type: oracle` to avoid context pollution:

```
Task tool parameters:
  subagent_type: "oracle"
  description: "Review implementation plan"
  prompt: |
    Review this implementation plan for issues.

    Plan file: /absolute/path/to/ai_docs/plans/YYYY-MM-DD-description.md
```

**Important:** Use absolute path for the plan file (e.g., `/home/user/project/ai_docs/plans/2025-01-15-feature.md`)

### Handling Directory-Based Plans

When the plan path is a directory (new structure):

1. **Detect plan format:**
   ```
   If path ends with `/` or is a directory → Directory-based plan
   If path ends with `.md` → Legacy single-file plan
   ```

2. **For directory-based plans, pass to Oracle:**

# Implementing Plan

## Overview

**You are the ORCHESTRATOR, not the implementer.**

Execute approved plans through skill chains: dispatch to subagents, validate output, manage phase boundaries.

```
NO IMPLEMENTATION COMPLETE WITHOUT IMPLEMENTATION-EXECUTOR SUBAGENT
```

BEFORE STARTING YOU MUST:
1. READ THE PLAN README.md
2. READ THE SUBAGENT-PROMPTS.md

3. Phase commit

# Running Automated Checks

## Overview

Run auto-fix, lint, typecheck, and tests in sequence. Report failures with structured output showing file:line:error for quick fixes.

## When to Use

- After implementing a feature or fix
- Before committing changes
- When user asks to "run tests", "check code", or "validate"
- After plan implementation phases

## When NOT to Use

- For exploratory code changes (run manually)
- When user explicitly says "skip tests"

## Steps

**Step 1: Discover Commands**
[WHY: Projects use different tooling; wrong commands waste time. Project docs define preferred commands]

Check files in this order (stop when you find commands):

1. **Project AI docs** (highest priority):
   - `AGENTS.md` or `CLAUDE.md` - AI-specific instructions

2. **Project docs**:
   - `README.md` - development/testing section
