# Reviewing Code

## Overview

You are the **orchestrator** for code reviews. Your job:
1. Determine **all** applicable review types
2. Gather context
3. Build structured input **per type**
4. Dispatch to code-review-agent — **one Task per type, in parallel**
5. Merge verdicts into combined report

**You do NOT review code yourself.** You dispatch to the agent.

## The Iron Law

```
NO "REVIEW COMPLETE" WITHOUT AGENT VERDICT
```

## Workflow Checklist

```
Code Review Orchestration:
- [ ] Step 1: Determine review types (plural — reason about changes)
- [ ] Step 2: Gather context
- [ ] Step 3: Build structured input per type
- [ ] Step 4: Dispatch to code-review-agent (one per type, in parallel)
- [ ] Step 5: Parse and merge agent verdicts
- [ ] Step 6: Handle combined result (fix or chain)
```

## Step 1: Determine Review Types

Select **ALL** applicable review types. This is NOT a cascading if/else — multiple types can and should be selected.

### Available Types

| Review Type | Covers | Triggers |
|-------------|--------|----------|
| `plan_compliance` | Implementation matches approved plan | Called from `implementing-plan` |
| `quality` | SOLID, functional programming, code smells, clean code | **Always included** (baseline) |
| `bug_fix` | Root cause fixed, regression test exists | Change metadata contains "fix" or "bug" |
| `refactoring` | Behavior preserved, no functional changes | Change metadata contains "refactor" |
| `feature` | Requirements, architecture, test coverage | New feature / capability added |
| `performance` | Benchmarks, profiling, trade-offs | Change metadata contains "perf" or "optimize" |
| `security` | STRIDE threat model, OWASP, high-confidence vulns | Called via /security-review, OR changes touch auth/crypto/secrets/validation/API boundaries |

### Selection Logic

```
1. Select primary type from context:
   - From implementing-plan → plan_compliance
   - From /security-review → security
   - Else match change metadata: bug_fix, refactoring, performance, feature

2. ALWAYS add quality as a SEPARATE dispatch (unless it IS the primary type)
   quality is the baseline review — it is never skipped.
   Do NOT rely on quality sections embedded in other type files (e.g.,
   plan-compliance.md's "Quality Review (Secondary)"). Those are summaries,
   not substitutes. The full quality.md (384 lines) must be dispatched
   as its own code-review-agent Task.

3. Reason about changed files to add more types:
   - security: IF changes touch authentication, authorization,
     cryptography, secrets, input validation, API boundaries,
     or file paths containing auth|security|crypto|token|session|password
   - performance: IF changes touch database queries, caching,
     connection pooling, hot paths, algorithms with complexity concerns,
     or file paths containing cache|pool|query|perf|benchmark

Result: Set of 2+ review types (primary + quality at minimum).
Exception: general review with no specific context → quality alone is valid.
```

### Example Selections

| Context | Selected Types |
|---------|---------------|
| From implementing-plan, normal code | `plan_compliance`, `quality` |
| From implementing-plan, auth code changed | `plan_compliance`, `quality`, `security` |
| Standalone bug fix | `bug_fix`, `quality` |
| Bug fix touching query cache | `bug_fix`, `quality`, `performance` |
| /security-review | `security`, `quality` |
| General review | `quality` |

## Step 2: Gather Context

### For All Review Types

```bash
# Get files to review
git diff --name-only HEAD~1

# Or for specific commit range
git diff --name-only <base>..<head>

# Get project standards
cat CLAUDE.md 2>/dev/null || cat $HOME/.claude/docs/code-principles.md
```

### For plan_compliance

```bash
# Load the plan
cat ai_docs/plans/<plan-file>.md

# Extract requirements (lines with - [ ] or - [x])
grep -E '^\s*-\s*\[' ai_docs/plans/<plan-file>.md
```

### For bug_fix

- Ticket/issue description
- Root cause explanation
- Expected regression test location

### For refactoring

- What improvement is claimed
- Which tests should pass unchanged

### For performance

- Benchmark results (before/after)
- Environment information

## Step 3: Build Structured Input (Per Type)

Build **one input per selected review type**. Each input follows the same format from `agents/code-review-agent/INPUT-OUTPUT.md`, but loads type-specific instructions.

**For each type in selected_types:**

```markdown
# Code Review Request

## Metadata
- **Review ID:** [generate-uuid or timestamp]-[review_type]
- **Review Type:** [this specific type]
- **Timestamp:** [ISO-8601]

## Files to Review
- [file1.ts]
- [file2.ts]

## Review Type Instructions
[Load contents from: agents/code-review-agent/types/{review_type}.md]

## Context

### Plan (only for plan_compliance)
**Path:** [plan file path]

**Requirements:**
[Copy requirements from plan]

**Success Criteria:**
[Copy success criteria from plan]

### Project Standards
**Source:** [CLAUDE.md or fallback]

```
[Contents of standards]
```

### Change Metadata
- **Ticket:** [if available]
- **Description:** [commit message or PR description]
- **Commit Range:** [if available]
```

Files to Review and Context sections are **shared** across all inputs. Only Metadata (Review Type) and Review Type Instructions differ per input.

## Step 4: Dispatch to Agent (Parallel)

Dispatch **all review types in a single message** so they run in parallel:

```python
# All Task() calls in ONE message — parallel execution
for review_type in selected_types:
  Task(
    subagent_type="code-review-agent",
    model="opus",
    description="Code review: {review_type}",
    prompt=structured_input_for[review_type]
  )
```

**Wait for ALL agents to return structured output.**

Each agent receives a single-type review request and returns a single-type report. The orchestrator (you) merges them in Step 5.

**Why parallel, not sequential:** Parallel dispatch has the same wall-clock time as a single dispatch. Sequential dispatch tempts early termination ("first failed, stop"), which skips reviews and misses issues in other dimensions. Always dispatch all types together.

## Step 5: Parse and Merge Agent Verdicts

From **each** agent's output, extract:

1. **Overall Verdict:** `APPROVED` or `NEEDS_FIXES`
2. **Compliance Verdict:** `COMPLIANT`, `NOT_COMPLIANT`, or `N/A`
3. **Quality Verdict:** `APPROVED` or `NEEDS_FIXES`
4. **Issues:** Count of Critical, Major, Minor
5. **Required Before Merge:** List of blocking issues

### Merge Rules

```
Combined Overall Verdict:
  NEEDS_FIXES if ANY agent returned NEEDS_FIXES
  APPROVED only if ALL agents returned APPROVED

Compliance Verdict:
  From plan_compliance agent (or N/A if not dispatched)

Quality Verdict:
  From quality agent (or N/A if not dispatched)

Issues:
  Union of all issues from all agents
  Deduplicate by file:line (same location, same issue = count once)

Required Before Merge:
  Union of all blocking items from all agents
```

### Per-Review Summary

Build a per-review verdict table:

```
| Review Type | Verdict | Issues |
|-------------|---------|--------|
| plan_compliance | COMPLIANT | 0 Critical, 0 Major |
| quality | NEEDS_FIXES | 1 Critical, 2 Major |
| security | APPROVED | 0 Critical, 1 Minor |
```

## Step 6: Handle Combined Result

### If ALL APPROVED

```
Code Review Complete
====================
Reviews: [list of types dispatched]
Overall: APPROVED

Per-Review Verdicts:
  plan_compliance: COMPLIANT ✅
  quality: APPROVED ✅
  [security: APPROVED ✅]

CHAIN: Invoking validating-implementation skill...
```

Then invoke:
```
Skill(validating-implementation)
```

### If ANY NEEDS_FIXES

Present combined issues from ALL agents (don't invoke validation yet):

```
Code Review: NEEDS_FIXES
========================
Reviews: [list of types dispatched]

Per-Review Verdicts:
  plan_compliance: COMPLIANT ✅
  quality: NEEDS_FIXES ❌
  [security: APPROVED ✅]

## Required Before Merge

### Critical Issues
[List from ALL agent outputs with file:line, fix, and source review type]

### Major Issues
[List from ALL agent outputs with file:line, fix, and source review type]

## Recommended (Non-blocking)
[Minor issues from ALL agents]

---
Fix the required issues, then I'll re-run the review.
```

**After fixes:**
1. Stage and commit fixes
2. Re-run this skill from Step 3 (rebuild inputs with new files)
3. Dispatch again (all types — not just the failing ones)

### Feedback Loop

```
Build inputs → Dispatch (parallel) → Parse all verdicts → Merge
                                                            ↓
                                               ALL APPROVED → Chain to validation
                                                            ↓
                                             ANY NEEDS_FIXES → Present combined issues
                                                            ↓
                                                        Fix issues
                                                            ↓
                                              Re-build inputs, re-dispatch all ←─┘
```

## Integration with implementing-plan

When called from `implementing-plan` (final review):

1. Review types: `plan_compliance` + `quality` (minimum)
   - Also `security` if changes touch auth/crypto/secrets/validation/API boundaries
   - Also `performance` if changes touch queries/caching/hot paths
2. Plan file path should be passed as context
3. After ALL reviews APPROVED, chain to `validating-implementation`

```
implementing-plan (after ALL phases)
       ↓
reviewing-code (this skill) → dispatches MULTIPLE code-review-agents in parallel
       ↓ (all APPROVED)
validating-implementation
```

## Standalone Usage

For reviews not part of implementing-plan:

1. User invokes `/review-code` or asks for review
2. Determine review types from context (Step 1 selection logic applies)
3. Follow same flow (parallel dispatch, merge verdicts)
4. After ALL APPROVED, present summary (no validation chain needed)

## Red Flags

- Reviewing code yourself instead of dispatching to agent
- Dispatching only ONE review type (except when quality is the sole type for a general review)
- Skipping the quality review — quality is ALWAYS included
- Skipping Step 3 (building structured input)
- Not loading the review type instructions
- Marking complete without APPROVED verdict from ALL dispatched agents
- Not chaining to validation after APPROVED (when from implementing-plan)
- Re-dispatching only failing types after fixes (re-dispatch ALL types)

## Common Rationalizations

| Rationalization | Why It's WRONG | Required Action |
|-----------------|----------------|-----------------|
| "This is simple, I'll review it myself" | Agent has comprehensive checklists you'll miss | **Always dispatch to code-review-agent** |
| "One review type is enough" | Each type covers non-overlapping concerns. plan_compliance doesn't check SOLID. quality doesn't check plan adherence. | **Select ALL applicable types** |
| "Quality review is redundant with plan_compliance" | plan_compliance checks adherence to plan. quality checks SOLID, FP, code smells, clean code — entirely different checklists (384 lines). | **Always include quality** |
| "I'll just dispatch the primary type to save time" | Parallel dispatch costs no extra wall-clock time. Skipping types means missed issues. | **Dispatch all types in parallel** |
| "Only the failing review needs re-dispatch" | Fixes may introduce new issues in other review dimensions | **Re-dispatch ALL types after fixes** |
| "Agent output looks fine" | Parse and verify the verdict explicitly | **Extract Overall Verdict from Summary table** |
| "Issues are minor, let's proceed" | Minor issues compound. Fix or document. | **Fix required issues before proceeding** |
| "I'll chain to validation now" | Only after ALL agents return APPROVED | **Wait for ALL agents APPROVED before chaining** |

# Committing Changes

Create git commits for changes made during this session.

## Process

1. **Analyze changes:**
   - Run `git status` and `git diff` to understand modifications
   - Consider whether changes should be one or multiple logical commits

2. **Plan commits:**
   - Group related files together
   - Draft clear commit messages in imperative mood
   - Focus on why changes were made

3. **Execute upon confirmation:**
   - Use `git add` with specific files (never `-A` or `.`)
   - Create commits with `git commit -n -m "message"`
   - Show result with `git log --oneline -n [N]`

## Critical Rules

- **NEVER add Claude attribution** - no co-author lines, no "Generated with Claude"
- Write commit messages as if the user wrote them
- Keep commits atomic and focused
