# Implementing Plan

## Overview

**You are the ORCHESTRATOR, not the implementer.**

Execute approved plans through skill chains: dispatch to subagents, validate output, manage phase boundaries.

**Core principle:** If you're reading/writing code directly, you're violating the pattern. Dispatch always.

## The Iron Law

```
NO IMPLEMENTATION COMPLETE WITHOUT IMPLEMENTATION-EXECUTOR SUBAGENT
```

BEFORE STARTING YOU MUST:
1. READ THE PLAN README.md
2. READ THE SUBAGENT-PROMPTS.md

FOR EACH **phase** YOU MUST FOLLOW THE STEPS:
1. Dispatch implementation-executor subagent (all tasks in phase)
2. Validate agent output
3. Phase commit
4. Phase tests (`Skill(running-automated-checks)`)

After **ALL phases**:
5. Final review (`Skill(reviewing-code)`)
6. Validation (`Skill(validating-implementation)`)

**Non-code tasks** (config/docs only) use lighter chain: Implement → Commit

---

## Core Constraints

NEVER skip reading the README.md for the plan.

NEVER skip reading the SUBAGENT-PROMPTS.md for the subagent.

NEVER implement tasks yourself - always dispatch to implementation-executor subagent.

NEVER skip review or validation because "tests pass" - the chain is non-negotiable.

NEVER redesign mid-implementation - trust the plan, document concerns in learnings.

NEVER show "Implementation Complete" without review AND validation verdicts.

**Violating the letter of these rules is violating the spirit of these rules. STOP and START OVER.**

## Process

### Step 0: Load Plan

1. **Determine plan structure:**
   - Directory → Read `README.md` for overview
   - Single `.md` file → Legacy plan

2. **Identify phases and status:**
   ```
   ai_docs/plans/YYYY-MM-DD-description/
   ├── README.md           # Overview, phase status
   ├── phase-1-[name].md   # Load when implementing phase 1
   └── phase-N-[name].md   # Load when implementing phase N
   ```

3. **Load only current phase file.** Don't load all at once - wastes context.

### Step 1: Execute Phase

**For each PHASE, dispatch ONE implementation-executor subagent:**

```python
Task(
  subagent_type="implementation-executor",
  model="opus",
  description=f"Implement Phase {phase_number}: {phase_name}",
  prompt=f"""
    Phase: {phase_number} - {phase_name}
    Plan file: {plan_file_path}

    Execute all tasks in this phase using TDD:
    1. For each task: RED → GREEN → REFACTOR
    2. Update plan file checkboxes
    3. Capture learnings
    4. Return Phase Execution Report
  """
)
```

**CRITICAL:**
- ONE subagent per PHASE, not per task
- Subagent handles ALL tasks internally
- You validate output and manage phase boundaries

### Step 1a: Validate Agent Output

Parse Phase Execution Report. Check:

| Check | Pass Condition | Fail Action |
|-------|----------------|-------------|
| Status | "PASS" in Summary | Re-dispatch with error context |
| TDD Results | RED+GREEN PASS for each task | Re-dispatch with failure details |
| Plan Updates | All checkboxes marked | Re-dispatch to update |

**After 3 failures → escalate to user** with attempt history.

### Step 2: Phase Commit and Tests

After validation passes:

```bash
git commit --no-verify -m "feat: [phase summary]"
```

Then run tests:
```
Skill(running-automated-checks)
```

If tests fail: fix, amend commit, re-run tests.

### Step 3: Update Dashboard

Edit README.md after each phase:
- Update phase status to "complete"
- Add learnings from implementation

### Step 4: Final Review and Validation

**When ALL phases complete:**

```
╔═══════════════════════════════════════════════════════════════╗
║  STOP - YOU ARE NOT DONE YET                                   ║
║                                                                 ║
║  "Tests pass" ≠ "Implementation complete"                      ║
║                                                                 ║
║  You MUST still run:                                           ║
║  1. Skill(reviewing-code)                                      ║
║  2. Skill(validating-implementation)                           ║
║                                                                 ║
║  Do NOT show summary to user until BOTH complete.              ║
╚═══════════════════════════════════════════════════════════════╝
```

1. `Skill(reviewing-code)` → Wait for APPROVED/FIXES
2. If FIXES: fix, amend, re-run review
3. Review APPROVED → `Skill(validating-implementation)`
4. Validation passes → NOW show summary

## Trust the Plan

**The plan is your approved roadmap. Execute it, don't redesign it.**

### What This Means

| Trust the Plan | Don't Trust Blindly |
|----------------|---------------------|
| Follow the plan's approach | Fix actual bugs when they appear |
| Stay within scope boundaries | Adapt when reality differs slightly |
| Use patterns the plan specifies | Handle edge cases within scope |
| Don't research alternatives | Debug issues intelligently |

### When to Escalate (Not Redesign)

If you encounter:
- Plan code has fundamental flaw (not just typo)
- Required API doesn't exist
- Approach is incompatible with reality

**STOP. Report to user. Do NOT redesign.**

```
ESCALATION:
Task: [task name]
Issue: [what's wrong]
Evidence: [error message, missing file]

The plan may need adjustment. How would you like to proceed?
```

### Missing Dependencies

If plan code references something that doesn't exist (like a `delay` function):

1. Check if it's defined elsewhere in plan
2. Check if it exists in codebase (grep)
3. If truly missing: **add minimal implementation** (utility function)
4. Document in learnings

**This is NOT redesign.** Adding a missing utility to make plan code work is implementation, not architecture change.

## Context Preservation

**Main context is for USER DISCUSSION and PHASE TRACKING only.**

| Main Context | Subagents |
|--------------|-----------|
| Phase progress | All implementation |
| User checkpoints | TDD execution |
| Escalations | File reading/writing |
| Final summary | Code review |

### Red Flags (Context Pollution)

If in main context you're doing ANY of:
- Reading files not in plan task
- Using Grep/Glob to search
- Running investigation commands
- Writing implementation code

**STOP → Dispatch to subagent → Only summary returns**

# Running Automated Checks

## Overview

Run auto-fix, lint, typecheck, and tests in sequence. Report failures with structured output showing file:line:error for quick fixes.

## When to Use

- After implementing a feature or fix
- Before committing changes
- When user asks to "run tests", "check code", or "validate"
- After plan implementation phases

## When NOT to Use

- For exploratory code changes (run manually)
- When user explicitly says "skip tests"

## Steps

**Step 1: Discover Commands**
[WHY: Projects use different tooling; wrong commands waste time. Project docs define preferred commands]

Check files in this order (stop when you find commands):

1. **Project AI docs** (highest priority):
   - `AGENTS.md` or `CLAUDE.md` - AI-specific instructions

2. **Project docs**:
   - `README.md` - development/testing section

3. **Build files** (fallback):
   - `Makefile` - targets (lint, test, typecheck)
   - `package.json` - `scripts` section
   - `Cargo.toml` - Rust → `cargo test`, `cargo clippy`
   - `pyproject.toml` - Python → pytest, ruff, mypy

**Step 2: Run Auto-fix FIRST**
[WHY: Most lint errors (semicolons, quotes, indentation) are auto-fixable. Running auto-fix first prevents wasting time manually editing errors that tools fix instantly]

```bash
npm run lint:fix    # or: make lint-fix, cargo fmt, ruff format
```

NEVER manually edit lint/format errors before running auto-fix.

**Step 3: Run Lint**
[WHY: Static analysis catches bugs, not just style]

```bash
npm run lint        # or: make lint, cargo clippy
```

**Step 4: Run Type Check**
[WHY: Type errors indicate logic bugs]

```bash
npm run typecheck   # or: tsc --noEmit, mypy, cargo check
```

**Step 5: Run Tests**
[WHY: Full suite catches integration issues]

```bash
npm test            # or: make test, cargo test, pytest
```

**Step 6: Report Results**
[WHY: Structured output enables quick navigation to errors and traceability of what ran and why]

Output a **markdown report** with these sections:

```markdown
## Automated Checks Report

### Commands Discovered

| Step | Command | Source | Why |
|------|---------|--------|-----|
| Auto-fix | `npm run lint:fix` | package.json | Fix auto-fixable lint/format errors before manual review |
| Lint | `npm run lint` | CLAUDE.md | Static analysis catches bugs beyond style |
| TypeCheck | `tsc --noEmit` | package.json | Type errors indicate logic bugs |
| Tests | `npm test` | package.json | Full suite catches integration issues |

### Execution Results

| Step | Command | Status |
|------|---------|--------|
| Auto-fix | `npm run lint:fix` | ✓ PASS |
| Lint | `npm run lint` | ✗ FAIL |
| TypeCheck | `tsc --noEmit` | ✓ PASS |
| Tests | `npm test` | ✓ PASS (15 passed) |

### Errors

#### Lint — 2 errors

| File | Line | Error |
|------|------|-------|
| src/foo.ts | 42 | 'x' is declared but never used |
| src/bar.ts | 10 | Missing return type annotation |

> Showing 15 of 47 errors

### Status: NEEDS FIXES
```

**Report rules:**
- **Commands Discovered**: List every command found in Step 1, its source file, and why it's run
- **Execution Results**: Show the exact command executed and its pass/fail status
- **Errors**: One subsection per failing step, max 15 errors each. If more exist, note total count
- **Status**: `READY` (all pass) or `NEEDS FIXES` (any fail)
- If a step was skipped (command not found), show `⊘ SKIPPED — no command found` in the results table

Parse file:line:error from tool output:
- ESLint: `file:line:col: message`
- TypeScript: `file(line,col): error`
- Jest: `at Object.<anonymous> (file:line:col)`
- Pytest: `file:line: message`

## Quick Reference

| Step | Action | On Fail |
|------|--------|---------|
| Discover | Check AGENTS.md/CLAUDE.md → README.md → Makefile/package.json | - |
| Auto-fix | `npm run lint:fix` or project-specific | Continue to lint |
| Lint | `npm run lint` or project-specific | Parse errors → table (max 15) |
| TypeCheck | `npm run typecheck` or project-specific | Parse errors → table (max 15) |
| Tests | `npm test` or project-specific | Parse failures → table (max 15) |
